# パイプライン監査スキル (pipeline-audit)

コーヒーブログのAI記事生成パイプライン全体を読み取り、アーキテクチャの説明と改善提案を日本語でレポートします。

## 実行手順

### Step 1: 現状コードの収集（並行実行）

以下を並行して読み取ってください：

```
Read: scripts/generate-post.ts
Read: scripts/generate-from-matrix.ts
Read: scripts/post-to-note.ts
Read: package.json
Read: topics-matrix.json
Read: topics.json（存在する場合）
Read: generated.json（存在する場合）
Read: .github/workflows/generate-post.yml
Read: .github/workflows/ci.yml
Glob: scripts/**/*.ts（他のスクリプトを把握）
Glob: docs/**/*.md（設計メモ・戦略ドキュメントを把握）
```

### Step 2: パイプラインフローの構造化

収集した情報から以下を把握してください：

- **トリガー**: 記事生成をどこで/何が起動するか（GitHub Actions スケジュール / 手動 / CLI）
- **AI生成**: 使用モデル・プロバイダー・プロンプト構成・出力形式
- **コンテンツ管理**: topics.json / topics-matrix.json の役割・規模・消費状況
- **出力**: 生成ファイルの保存先・形式・命名規則
- **投稿**: note.com への投稿フロー（draft/publish の扱い）
- **品質ゲート**: レビュー・リーガルチェック・重複防止の仕組み（あれば）
- **記録管理**: generated.json の役割と消費済み組み合わせ数

### Step 3: 課題の特定

以下の観点で現状を評価してください：

**可用性・スケーラビリティ**
- GitHub Models (GPT-4o) の利用制限リスク
- matrix 組み合わせ数と消費速度のバランス
- 並列生成・バッチ処理の有無

**品質管理**
- 生成後の自動レビュー有無
- 重複コンテンツの防止機構
- 投稿前の人間レビューフロー

**運用性**
- CI/CDの網羅性（lint/typecheck/test）
- エラー時の通知・リトライ機構
- 生成履歴の可視性

**拡張性**
- 画像統合（Pexels API 等）の実装状況
- 複数プラットフォーム対応
- コスト最適化（モデル選択・キャッシュ）

### Step 4: レポート生成

以下のフォーマットで出力してください：

---

## 出力フォーマット

```markdown
# パイプライン監査レポート
> 生成日: {今日の日付}

## 1. パイプライン全体フロー

```
[ASCII フロー図: トリガー → 生成 → 保存 → 投稿]
```

## 2. コンポーネント別説明

| コンポーネント | ファイル | 役割 | 現状 |
|-------------|---------|------|------|

## 3. スケール感

- 総コンテンツ候補数: X件（matrix: 豆×抽出×切り口）
- 生成済み: X件（Y%消費）
- 週X本ペースで約X年分のストック

## 4. 現状の課題

### 🔴 高優先度
- 課題1: 説明

### 🟡 中優先度
- 課題2: 説明

### 🟢 低優先度 / 将来対応
- 課題3: 説明

## 5. 改善提案（優先度順）

### P1（今すぐ着手推奨）
**[提案タイトル]**
- 現状の問題: ...
- 提案内容: ...
- 期待効果: ...
- 実装コスト: 小/中/大

### P2（次のスプリント）
...

### P3（将来対応）
...

## 6. まとめ

[3〜5文で総括。強みを認めた上で最重要アクションを1つ明示]
```

---

## 注意事項

- ファイルが存在しない場合はそのコンポーネントが「未実装」として扱う
- generated.json からコンテンツ消費率を実際に計算する
- topics-matrix.json の beans × methods × angles の総数を実際に計算する
- 改善提案は「具体的に何をどのファイルに追加するか」まで踏み込む
- レポートは会話内に出力するだけでよい（ユーザーが保存を求めた場合のみ docs/ に書き出す）
