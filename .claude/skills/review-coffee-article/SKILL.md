# コーヒー記事 網羅的レビュースキル

生成されたコーヒー記事を8つの観点で網羅的にレビューし、公開品質かどうかを判定する。
スコアリング・指摘・改善提案を日本語で出力し、`--fix` 指定時は自動改善する。

## 使い方

```
/review-coffee-article [ファイルパス] [--fix] [--strict]
```

**引数:**
- `[ファイルパス]`: レビュー対象ファイル（省略時は `output/` 内の最新ファイルを使用）
- `--fix`: レビュー後に問題箇所を自動改善してファイルを上書き保存
- `--strict`: 厳格モード（合格ラインを 75→85 に引き上げ。公開直前の最終確認に使用）

---

## 実行手順

### Step 0: ファイル特定

1. ファイルパスが指定されていれば Read ツールで読み込む
2. 省略時は `output/` ディレクトリを Glob で検索し、最新の `.md` ファイルを対象とする
3. ファイルの frontmatter（`topic` / `category` / `date`）を読み取り、レビュー基準を決定する
   - `category` が不明な場合は記事本文から推定する

---

### Step 1: 8軸スコアリング

以下の8軸でそれぞれ評価し、**軸ごとのスコア・判定理由・具体的指摘**を出力する。

---

#### Axis 1: 事実の正確性（ファクトチェック） /15

チェック項目:
- [ ] 産地・地名・農園名のスペル・表記が正しいか
- [ ] 標高・気温・収穫時期などの数値が妥当な範囲か（例: エチオピア イルガチェフェ = 1750〜2200m）
- [ ] 精製方法・焙煎理論・抽出数値（温度・比率）が一般的知見と矛盾しないか
- [ ] 健康に関する主張が過度・根拠なしになっていないか
- [ ] 誤解を招く断定的表現（「必ず」「絶対に」「最高」）がファクトなしに使われていないか

採点基準:
- 13〜15: 事実に誤りなし、数値も適切
- 9〜12: 軽微な不正確さ（表記ゆれ・数値が若干ずれ）
- 5〜8: 誤情報が1〜2箇所、修正が必要
- 0〜4: 重大な事実誤認、または根拠なき断定が複数

---

#### Axis 2: コーヒー専門性 /15

チェック項目:
- [ ] カテゴリに応じた専門知識が記事に示されているか
  - 豆・産地: テロワール・品種・精製方法の言及
  - 抽出・レシピ: 抽出理論（TDS・収率）・SCA基準への意識
  - 道具・機器: スペック・メカニズムの正確な説明
  - コーヒー文化: 歴史的背景・ウェーブ理論・社会的文脈
- [ ] 専門用語が初出時に平易な説明を伴っているか
- [ ] 読者が「これはプロが書いた」と感じられる洞察があるか
- [ ] 表面的なwikiレベルの説明で止まっていないか

採点基準:
- 13〜15: 深い専門知識、独自の視点・洞察あり
- 9〜12: 適切な専門知識、標準的な深度
- 5〜8: 知識は浅め、ありきたりな記述が目立つ
- 0〜4: 専門性がなく、一般知識の羅列

---

#### Axis 3: 読者エンゲージメント /15

チェック項目:
- [ ] タイトル: 30文字以内か、数字・疑問形・「〜の理由」など引きが強いか
- [ ] リード文: 冒頭3〜4文で共感・驚き・疑問のいずれかを使って引き込んでいるか
- [ ] 読み飛ばし耐性: 見出し（##）だけ読んでも記事の流れが把握できるか
- [ ] 締め: 読後に「試したい」「行きたい」「買いたい」感情が湧くか
- [ ] 語りかけ: 読者への直接表現（「〜してみてください」「〜ですよね」）が1記事2回以内に抑えられているか
- [ ] 締めが1文で完結しているか（二重締め・言い直しがないか）

採点基準:
- 13〜15: 強い引き、自然な流れ、行動喚起が明確
- 9〜12: 及第点、一部改善余地あり
- 5〜8: フックが弱い、または締めが尻切れ
- 0〜4: 読む動機が生まれない構成

---

#### Axis 4: 構成・フォーマット /10

チェック項目:
- [ ] 文字数がカテゴリ基準を満たすか
  - 豆・産地: 1800〜2000字
  - 抽出・レシピ: 1600〜2000字
  - カフェ情報: 1200〜1500字
  - 道具・機器: 1600〜2000字
  - コーヒー文化: 1800〜2000字
- [ ] 見出し（##）の数・配置が適切か（3〜6個が目安）
- [ ] 箇条書き・番号リストが適切に使われているか（特にレシピ系）
- [ ] 一段落の長さが読みやすいか（目安: 3〜5文）
- [ ] 文体が「です・ます」調で統一されているか

採点基準:
- 9〜10: 文字数・構成・文体すべて適切
- 6〜8: 軽微な逸脱（文字数が±200字程度）
- 3〜5: 文字数が大きく外れる、または構成が崩れている
- 0〜2: 構成が成立していない

---

#### Axis 5: 具体性・信頼性 /10

チェック項目:
- [ ] 曖昧表現（「ある程度」「いくつか」「〜かもしれません」）が残っていないか
- [ ] 数値・固有名詞・エピソードが各セクションに1つ以上含まれているか
- [ ] 比較や評価が相対的（「最高」「一番」）ではなく根拠付きか
- [ ] レシピ系の場合: 温度・量・時間の数値が明記されているか
- [ ] 引用・参照元が明示または暗示されているか

採点基準:
- 9〜10: 具体性が高く信頼感がある
- 6〜8: 一部に曖昧表現あり
- 3〜5: 曖昧表現が目立ち、読者が行動に移しにくい
- 0〜2: ほとんどが抽象・一般論

---

#### Axis 6: note.com最適化 /10

チェック項目:
- [ ] 冒頭140文字（プレビュー）が単独で意味をなし、読者の興味を引くか
- [ ] カテゴリに合ったハッシュタグ5個以上が末尾に付いているか
- [ ] `[画像: 〇〇]` の挿入位置マークが適切な箇所（各セクション間目安）に入っているか
- [ ] note.com固有のフォーマット制限（極端な改行連打・HTML等）を侵していないか
- [ ] タイトルがnote.comの検索・発見に適した表現か

採点基準:
- 9〜10: note.com投稿ですぐ使えるレベル
- 6〜8: 軽微な調整で対応可能
- 3〜5: プレビューまたはハッシュタグに問題
- 0〜2: note.com対応が不十分

---

#### Axis 7: SEO・拡散可能性 /15

チェック項目:
- [ ] タイトルにメインキーワード（産地名・器具名・手法名など）が入っているか
- [ ] 見出しにサブキーワードが自然に含まれているか
- [ ] 「保存したくなる」要素があるか（チェックリスト・比較表・まとめ箇所）
- [ ] 「シェアしたくなる」独自の切り口・驚きがあるか
- [ ] 長尾キーワード（「〇〇 入門」「〇〇 やり方」など）への対応

採点基準:
- 13〜15: SEO・シェア両面で強い
- 9〜12: キーワードは入っているが拡散要素が弱い
- 5〜8: タイトル・見出しにキーワードが不足
- 0〜4: SEO・拡散要素がほぼない

---

#### Axis 8: 倫理・コンプライアンス /10

**重要:** このレビューより前に `/legal-check-coffee-article` が実行済みであること。
リーガルチェック未実施の場合は Axis 8 を採点せず、先にリーガルチェックを実行するよう案内する。

チェック項目:
- [ ] `/legal-check-coffee-article` の実行済みスコアが 75/100 以上か（記録があれば参照）
- [ ] 健康効果の過度な主張（「飲めば〇〇が治る」等）がないか（薬機法・健康増進法）
- [ ] 価格・営業時間には「公式サイトでご確認ください」が付いているか（景品表示法）
- [ ] 特定の個人・ブランドへの根拠なき誹謗中傷がないか（名誉毀損・不正競争防止法）
- [ ] 著作物の無断引用・コピーペーストの痕跡がないか（著作権法）
- [ ] PR・提供品がある場合、冒頭に開示表記があるか（ステルスマーケティング規制）
- [ ] 医療的・栄養的アドバイスの代替として誤解される表現がないか

採点基準:
- 9〜10: リーガルチェック 90+ かつ問題なし
- 7〜8: リーガルチェック 75〜89 かつ軽微な表現修正で対応済み
- 4〜6: リーガルチェック未実施 または 要確認の主張が含まれる → 先にリーガルチェックを実行
- 0〜3: 倫理・法的リスクのある記述あり、または リーガルチェック 60 未満

---


**判定基準（通常モード）:**
| スコア | 判定 | アクション |
|--------|------|------------|
| 85〜100 | ✅ 公開可 | そのまま投稿可 |
| 70〜84 | ⚠️ 要軽微修正 | 指摘箇所を修正後に投稿 |
| 55〜69 | 🔧 要構成改善 | 見出し・リード・締めを再構成 |
| 0〜54 | ❌ 再生成推奨 | `/generate-coffee-article` で書き直し |

`--strict` 指定時は合格ラインが **85→90** に引き上げ。

---

### Step 3: 詳細フィードバック出力

スコアにかかわらず、以下を必ず出力する:

```
## 🔴 必須修正（公開前に直す）
（致命的な問題のみ。なければ「なし」）

## 🟡 推奨改善（品質向上）
（改善すると読者体験・SEOが上がる箇所）

## 🟢 良かった点
（次回の記事生成でも活かすべき優れた表現・構成）

## ✏️ 具体的な修正例
（「この文を〇〇に変える」という具体例を1〜3個）
```

---

### Step 4: 自動改善（--fix 指定時のみ）

1. 🔴 必須修正を優先的に適用する
2. 🟡 推奨改善のうち機械的に適用できるものを適用する
   - タイトルの文字数超過 → 短縮
   - ハッシュタグ不足 → カテゴリ基準のタグを追加
   - `[画像: 〇〇]` 不足 → 適切な位置に追加
   - 冒頭140文字のプレビュー弱化 → リード文を微調整
3. 改善後のファイルを同名で上書き保存する
4. 改善前後の差分サマリーを出力する
